02 / EN 2 MINUTES

L'essentiel.

1

Quoi

Agent IA qui formalise le devoir de conseil du courtier. Il recoit un besoin client, extrait le profil CRM, identifie la branche d'assurance, execute 3 blocs IA (collecte, diagnostic, rapport), et genere un rapport de conseil pre-rempli conforme FSMA (Belgique) ou ACPR V2 (France).

2

Pourquoi

Un courtier passe ~2h par rapport de conseil, 15+ rapports/semaine = 30h de paperasse. La qualite est inegale, les references reglementaires manquent, les controles FSMA/ACPR trouvent des rapports incomplets → sanctions.

3

Combien & Quand

MVP 13 000€ HT (~30 jours) + PROD 7 000€ HT (~15 jours) = 20 000€ HT total. Module le plus complexe de BrokerONE.

4

Role dans BrokerONE

Augmented est le coeur de la boucle vertueuse. Il est alimente par Sentinel (besoin assurance detecte) et Checkup (lacune critique). Quand le courtier valide un dossier, Advice declenche automatiquement une sollicitation de satisfaction — timing optimal.

PART 1 / LE PROBLEME

Le devoir de conseil aujourd'hui.

La reglementation IDD (Belgique) et DDA (France) impose au courtier de documenter formellement chaque conseil en assurance. Situation du client, besoins identifies, produits recommandes, alternatives presentees, references reglementaires. Tout doit etre ecrit, signe, archive.

03 / CE QUI SE PASSE AUJOURD'HUI

Le quotidien du courtier.

📄

Templates Word manuels

Le courtier ouvre un template Word vierge, copie-colle les infos du client depuis le CRM, remplit manuellement chaque section. 1h30 a 2h par dossier. Erreurs de copie, oublis, incoherences.

Cout : 30h/semaine de paperasse pour un courtier actif.
⚠

Qualite inegale

Chaque courtier redige a sa facon. Certains sont exhaustifs, d'autres expedient. Les references reglementaires sont souvent absentes ou erronees. Pas de standard.

Risque : Inspection FSMA/ACPR → rapport juge non conforme.
🔍

References manquantes

Le rapport de conseil doit citer les articles de loi, circulaires FSMA, recommandations ACPR qui justifient le conseil donne. Qui connait par coeur l'article 279 de la Loi Assurances ?

Risque : Rapport sans base juridique = conseil non fonde.
🚫

Dossiers oublies

Un client appelle pour un devis, le courtier repond oralement. Le rapport de conseil ? Jamais redige. Si le client se plaint 3 ans plus tard, aucune trace ecrite.

Risque : Responsabilite civile professionnelle du courtier engagee.

11 / BLOC 2 — ANALYSE & DIAGNOSTIC

Le diagnostic IA.

Le coeur d'Augmented. Claude Sonnet croise le profil client avec la branche d'assurance et la reglementation applicable. Il produit un diagnostic structure.

A

Croisement 3 dimensions

Profil client x branche d'assurance x reglementation applicable. L'IA ne genere pas du texte generique — elle analyse une situation precise dans un cadre reglementaire precis.

B

Detection des lacunes

Ce qui manque vs ce qui devrait etre couvert. Exemple : famille avec 2 enfants mais aucune RC familiale → lacune critique detectee. Proprietaire sans assurance vol → lacune identifiee.

C

Scoring des besoins

Chaque besoin identifie est score sur 3 axes : urgence (le client est-il expose maintenant ?), impact (quelle est la gravite potentielle ?), obligation reglementaire (est-ce legalement requis ?).

D

References reglementaires

Pour chaque besoin identifie, l'IA injecte les references : quel article FSMA, quelle circulaire, quelle recommandation ACPR. Le courtier n'a plus a chercher.

// Sortie structuree du Bloc 2
diagnostic: {
  besoins: [{ branche, description, urgence, impact, reglementaire, references[] }],
  lacunes: [{ type, description, gravite, couverture_actuelle, couverture_recommandee }],
  profil_risque: "modere" | "eleve" | "critique",
  reglementation_applicable: [{ code, libelle, article, source }]
}
12 / BLOC 3 — GENERATION & RAPPORT

Le rapport pre-rempli.

Le Bloc 3 transforme le diagnostic en un rapport de conseil formel, pret a etre valide et signe. Deux formats selon le pays : FSMA (Belgique) ou ACPR V2 (France).

Sections du rapport

1. Identification courtier + client
2. Resume de la situation du client
3. Besoins identifies et hierarchises
4. Produits recommandes (avec justification)
5. Alternatives presentees (obligatoire)
6. References reglementaires
7. Recommandation formelle du courtier
8. Date, signature, mentions legales

Formats de sortie

📄 HTML (in-app preview)

Apercu interactif dans le dashboard. Le courtier peut relire, annoter, modifier avant validation.

🗎 DOCX (telechargement)

Fichier Word genere via la librairie docx. Pret a imprimer, signer, archiver. En-tete personnalise avec le logo du courtier.

Personnalisation : en-tete, logo courtier, pied de page, couleurs — configurable dans les parametres du courtier.
PART 5 / HUMAN-IN-THE-LOOP

LA garantie architecturale.

C'est le principe fondateur d'Augmented. L'IA genere, le courtier decide. Aucune transmission automatique. Jamais. Le courtier est personnellement responsable de son conseil — c'est la loi.

13 / LE WORKFLOW DE VALIDATION

5 etapes, 0 automatisme.

1

L'IA genere le brouillon

Les 3 blocs s'executent. Le rapport est genere en mode brouillon. Statut : "draft". Visible uniquement par le courtier.

2

Le courtier relit

Le rapport s'affiche en apercu HTML. Le courtier lit chaque section, verifie les recommandations, s'assure que le diagnostic correspond a sa connaissance du client.

3

Le courtier modifie si necessaire

Chaque section du rapport est editable. Le courtier peut reformuler, ajouter un contexte, modifier une recommandation. L'IA n'est qu'un point de depart.

4

Le courtier valide explicitement

Checkbox obligatoire : "Je confirme avoir relu et approuve ce rapport de conseil". Bouton de validation. Rien ne peut etre transmis au client avant cette etape.

5

Le rapport est valide

Statut passe a "validated". L'export DOCX devient disponible. La validation est logguee : timestamp, broker_id, IP. Declenchement automatique d'Advice pour la sollicitation satisfaction.

14 / POURQUOI C'EST NON-NEGOCIABLE

Exigence reglementaire.

🇧🇪 Belgique

Art. 284 Loi Assurances : le courtier est personnellement responsable du conseil donne.
Circulaire FSMA_2022_06 : le conseil doit etre adapte aux besoins du client et documente.
Code deontologique Feprabel : le courtier ne peut pas deleguer sa responsabilite a un outil.

🇫🇷 France

Art. L521-4 Code des assurances : obligation de conseil personnalise et documente.
Recommandation ACPR 2024-R-03 : le rapport de conseil V2 est obligatoire.
L'intermediaire est responsable du conseil, pas l'outil.
En resume : L'IA est un assistant. Le courtier est le decideur. Le human-in-the-loop n'est pas un "nice to have" — c'est une obligation legale. Augmented respecte ca de bout en bout.

Audit trail complet

☑ Chaque validation logguee avec timestamp exact
☑ Broker ID du courtier qui a valide
☑ Hash du rapport au moment de la validation (integrite)
☑ Modifications trackees (diff entre draft IA et version validee)
☑ Retention 5 ans minimum (exigence FSMA/ACPR)

PART 15 / EDGE CASES PROD

Quand tout ne se passe pas bien.

Le MVP gere le happy path. La production gere les cas limites qui arrivent dans la vraie vie.

PROD

CRM indisponible

Brio ou BrokerCloud est en panne. Fix : Fallback saisie manuelle. Le courtier entre les donnees client a la main. Le rapport est genere quand meme.

PROD

Claude timeout

La generation de rapport prend trop de temps. Fix : Retry 3x avec backoff. Si echec total, rapport partiel sauvegarde + notification pour relancer.

PROD

Branche non reconnue

Le CRM renvoie une branche inconnue. Fix : Selection manuelle de la branche par le courtier. Mapping CRM → branches Augmented configurable.

PROD

Client avec 0 contrats existants

Nouveau client sans historique. Fix : Le bloc Diagnose adapte son analyse. Pas de comparaison avec l'existant, focus sur l'analyse des besoins purs.

PROD

Dossier multi-branche

Un client a besoin de 3 assurances en meme temps (auto + habitation + RC familiale). Fix : Dossier unique avec 3 sections de diagnostic. Rapport consolide avec table des matieres.

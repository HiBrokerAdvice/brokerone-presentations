03 / ENGINE.JS — LE MOTEUR IA FOURNI

Architecture de prompts 3 couches.

Le moteur IA de Bruno n'est pas un simple appel Claude. C'est une architecture de prompts en 3 couches avec prompt caching, detection automatique de branche, conversation multi-turn, et mode fast-track.

COUCHE 1 System prompt — Toujours charge

Le prompt systeme est toujours present. Il definit le role de l'IA (assistant devoir de conseil), les regles de base, le format de sortie attendu, et les contraintes de securite (human-in-the-loop obligatoire).

COUCHE 2 BLOC2 — Reference docs (caches) — ~50K tokens, toujours en cache

Les documents de reference fondamentaux sont toujours charges et caches via prompt caching Claude. 6 fichiers BLOC2 par pays. Cadre reglementaire general, structure des rapports, templates de sortie. Le caching evite de re-tokeniser ~50K tokens a chaque requete → cout et latence reduits.

COUCHE 3 BLOC3 — Branch injection (conditionnel) — Injecte a la demande selon la branche

Quand une branche est detectee (auto-detection ou saisie manuelle), le knowledge pack specifique est injecte. Chaque branche a son propre fichier BLOC3 : arbres de decision, templates KYC, checklists de conformite, references reglementaires specifiques. Seul le BLOC3 pertinent est charge — pas les 35 autres.

// Architecture engine.js (simplifie)
const messages = [
  { role: "system", content: SYSTEM_PROMPT },
  { role: "user", content: BLOC2_REFERENCE_DOCS, cache: true }, // ~50K tokens, cached
  { role: "user", content: getBLOC3(branche) }, // conditionnel
  ...conversationHistory, // multi-turn
  { role: "user", content: userMessage }
];

// Features
autoDetectBranch(clientProfile) // Detection automatique de branche
fastTrackMode() // Mode rapide sans conversation
multiTurnConversation() // Echanges courtier ↔ IA
server.js aussi fourni : API REST complete — create dossier, multi-turn message, get/validate dossier, verify, referentiels, branches. Tout le backend API est deja code.
supabase_client.js aussi fourni : CRUD complet, file upload, webhook notifications avec HMAC, human-in-the-loop validation lock.

07 / POURQUOI AUGMENTED APRES SENTINEL

Le chainon manquant.

Sentinel trie les emails. Quand il detecte un besoin d'assurance, il cree un dossier Augmented automatiquement. Sans Augmented, ce dossier reste vide — le courtier doit formaliser son conseil a la main. Augmented transforme chaque besoin detecte en rapport de conseil conforme, en quelques minutes au lieu de 2 heures.

⚖️
GARANTIE ARCHITECTURALE

L'IA propose, le courtier decide. Aucun document ne peut etre transmis au client sans validation explicite du courtier. C'est une contrainte dans le code ET dans la base de donnees — pas une option qu'on peut desactiver.

14 / HORS PERIMETRE

Ce qui n'est pas inclus.

✗ Yousign e-signature — Integration signature electronique des rapports (phase future). Le courtier imprime et fait signer pour l'instant.
✗ IA apprenant des corrections — L'IA ne s'ameliore pas automatiquement a partir des modifications du courtier. Pas de fine-tuning from corrections (phase ulterieure).
✗ Integration catalogue produits — Pas de connexion aux tarificateurs des compagnies d'assurance. Augmented formalise le conseil, il ne compare pas les prix.
✗ Multi-langue NL — Interface et rapports en francais uniquement pour cette phase. Neerlandais en phase future si demande.
✗ Branches exotiques — Maritime, aviation, agriculture — pas dans les knowledge packs actuels.

03 / LE PROBLÈME

Le chaos de la boîte mail.

Un courtier reçoit des dizaines d'emails par jour. Chaque message est lu, interprété, puis classé — manuellement. En pratique, des emails critiques passent entre les mailles. Chaque jour.

04 / 4 SCÉNARIOS RÉELS

Ce qui se passe aujourd'hui.

SCÉNARIO 1

Réclamation noyée

Vendredi 17h42. Mme Dupont envoie : "Je suis extrêmement mécontente du traitement de mon sinistre". L'email se retrouve entre 40 autres messages. Lundi, le courtier ne le voit pas. Mercredi, Mme Dupont contacte le Médiateur.

Risque FSMA/ACPR : Délai d'AR de 10 jours dépassé. Non-conformité. Mention au rapport de contrôle.
SCÉNARIO 2

Opportunité manquée

Un prospect demande un devis MRH par email. Le courtier est en rendez-vous toute la matinée. Il répond 3 jours plus tard — le prospect a déjà signé chez un concurrent en ligne en 10 minutes.

Coût : Commission annuelle perdue. Client jamais récupéré.
SCÉNARIO 3

Demande RGPD ignorée

M. Martin écrit : "Je souhaite savoir quelles données vous détenez sur moi". L'objet est vague : "question". Le courtier pense que c'est une question commerciale. 45 jours passent. M. Martin saisit la CNIL/APD.

Risque : Amende APD/CNIL. Délai légal de 1 mois dépassé.
SCÉNARIO 4

Ambassadeur invisible

Mme Laurent écrit : "Merci infiniment pour votre réactivité, je vous recommande déjà autour de moi". Le courtier lit, sourit, passe au suivant. Aucune action. Pas de demande d'avis Google. Ambassadeur perdu.

Coût : Avis Google jamais sollicité. Bouche-à-oreille non structuré.

08 / PIPELINE DÉTAILLÉ

Le trajet d'un email.

1

Arrivée de l'email

L'email arrive dans la boîte M365 ou Gmail du courtier. Rien ne change pour lui.

↓
2

Capture par Sentinel

Webhook (M365 Graph notifications) ou polling périodique (Gmail API). L'email est récupéré avec son contenu complet : sujet, corps, expéditeur, pièces jointes (métadonnées).

↓
3

Pré-filtrage automatique

Exclusion des newsletters (header List-Unsubscribe), noreply/no-reply/notification/mailer-daemon, spam/promotions (Gmail label, Graph folder Junk), et emails envoyés par le courtier lui-même.

↓
4

Classification hybride

Option A (recommandée) : Claude API avec prompt structuré. Option B (fallback) : regex + scoring JavaScript. Résultat : 1 des 4 catégories + score de confiance 0-1.

↓
5

Enrichissement CRM

Croisement avec le CRM si disponible (Brio/BrokerCloud). Le client est-il connu ? Quels contrats ? Historique des interactions ?

↓
6

Forward vers l'agent concerné

HTTP POST vers l'endpoint Advice correspondant (déjà codé). Le payload contient la catégorie, le score, le résumé IA, et toutes les métadonnées.

↓
7

Log dans scan_logs

Chaque scan est tracé : nb emails scannés, nb détections, répartition par catégorie, erreurs éventuelles. Traçabilité complète.

14 / CLASSIFICATION IA — 2 OPTIONS

IA hybride : le meilleur des deux mondes.

OPTION A RECOMMANDÉ

Claude API

• Prompt structuré avec définitions de catégories, listes de mots-clés, règles de combinaison
• Output JSON strict (catégorie, score, résumé, métadonnées)
• Comprend le contexte, l'ironie, les demandes implicites
• Multilingue : FR / NL / DE
• Coût : ~0.01€ / email
• Temps : ~1-2 secondes
OPTION B FALLBACK

Regex + Scoring JS

• N8N Function node
• Pattern matching + scoring par mots-clés
• Moins précis mais aucun coût API
• Utilisé quand : timeout Claude, quota dépassé, cas évidents
• Temps : <100ms
• Précision : ~75% vs ~95% pour Claude

Approche hybride (stratégie retenue)

1. Regex d'abord. Si confidence >90% → on utilise le résultat regex directement. Rapide, gratuit.
2. Si confidence <90% → appel Claude API pour analyse contextuelle. Plus lent, mais bien plus précis.
3. Si Claude timeout ou erreur → fallback sur le résultat regex + flag "vérification manuelle".
Résultat : ~60% des emails classés par regex seul (cas évidents). ~40% nécessitent Claude. Coût moyen réduit de 40%.

18 / CONNECTEUR MICROSOFT 365

Microsoft Graph API v1.0

ParamètreValeur
APIMicrosoft Graph v1.0
AuthOAuth 2.0 (Authorization Code ou Client Credentials)
ScopeMail.Read, Mail.ReadBasic
EndpointGET /me/messages?$filter=receivedDateTime ge {date}
Pagination@odata.nextLink, $top=50 (max 1000)
Mapping : id → email_id | subject → objet | from.emailAddress → expediteur | receivedDateTime → date_reception | bodyPreview + body.content → contenu
19 / CONNECTEUR GMAIL

Gmail API v1

ParamètreValeur
APIgmail.googleapis.com/gmail/v1
AuthOAuth 2.0, scope gmail.readonly
EndpointGET /users/me/messages?q=after:{date}
PaginationnextPageToken, maxResults=100
Particularités Gmail : L'endpoint retourne uniquement les IDs. Un 2e appel est nécessaire pour le détail. Le corps est en base64url dans payload.parts[].body.data (MIME multipart). Subject et From sont dans payload.headers[].

06 / MERGE V1 + V2 — ÉTAPE PAR ÉTAPE

6 étapes d'intégration.

1

Intégrer tes 3 endpoints backend dans l'app v1

Chaque endpoint reçoit des données (réclamation, demande RGPD ou ambassadeur), les valide avec Zod, et les stocke en base.

// Endpoint 1 : Réclamations
POST /api/v1/advice/reclamation
// Reçoit une réclamation (via API ou module futur)
// Validation Zod : émetteur, objet, niveau (1-4), délai AR/réponse

// Endpoint 2 : Demandes RGPD
POST /api/v1/advice/rgpd
// Reçoit les demandes de droits RGPD
// Validation Zod : type de droit, identité, restrictions LCB-FT

// Endpoint 3 : Ambassadeurs
POST /api/v1/advice/ambassador
// Reçoit les clients potentiellement ambassadeurs
// Validation Zod : segment, score sentiment, conversion Google
2

Intégrer les prompts IA (441 lignes)

Trois familles de prompts, chacune avec son rôle précis :

ACCUSÉ DE RÉCEPTION

Conforme ACPR 2024-R-02. Accuse réception de la réclamation sous 10 jours ouvrés. Mentionne le numéro de dossier, le délai max de traitement (2 mois), la possibilité de saisine du médiateur.

RÉPONSE 3 TONS

Génère 3 variantes de réponse pour chaque réclamation : ton professionnel, ton empathique, ton direct. Le courtier choisit et personnalise.

6 TEMPLATES RGPD

Accès aux données, portabilité, rectification/effacement/limitation, opposition, directives post-mortem, refus motivé. Cite les articles RGPD + recours CNIL ou APD.

3

Exécuter la migration SQL (4 nouvelles tables)

4 tables à créer dans Supabase, chacune avec ses RLS policies :

TABLE 1
reclamations_advice

Niveaux 1-4, délai AR (10j ouvrés) et délai réponse (2 mois), émetteur sensible (oui/non), saisine médiateur, statut (ouvert/en cours/clôturé), historique des échanges.

TABLE 2
demandes_droits_advice

8 types de droits RGPD, restrictions LCB-FT (anti-blanchiment), vérification d'identité (obligatoire), délai de traitement (1 mois extensible à 3), statut, pièces justificatives.

TABLE 3
ambassadeurs_advice

Segments PREMIUM / POTENTIEL / NEUTRE, score de sentiment (calculé par IA), taux de conversion Google Review, historique des sollicitations.

TABLE 4
audit_logs

Journal d'audit complet. Chaque appel vers les endpoints est tracé. Idempotence (pas de doublon si la même donnée arrive deux fois).

+ FONCTIONS SQL UTILITAIRES
get_reclamations_en_retard()
Liste les réclamations dont le délai AR ou réponse est dépassé
get_demandes_rgpd_en_retard()
Liste les demandes RGPD dont le délai de traitement est dépassé
get_ambassadeur_stats()
Statistiques agrégées par courtier (nombre, segments, conversions)
4

Intégrer les types TypeScript + validateurs Zod

Chaque endpoint, chaque table a son type TypeScript et son schéma Zod. Toute donnée entrante est validée avant d'atteindre la base. Pas de donnée malformée en production.

5

Intégrer la config dynamique FR/BE (reg-config.ts)

Selon le pays du courtier, le système adapte automatiquement : régulateur (ACPR vs FSMA), autorité protection données (CNIL vs APD), délais légaux, références articles de loi, médiateur compétent.

6

Tester que les endpoints répondent correctement

Vérifier chaque flux de bout en bout. Endpoint par endpoint, table par table, prompt par prompt.

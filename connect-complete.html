03 / LE PROBLÈME

Le courtier invisible.

Un courtier sans présence sociale n'existe pas pour les nouveaux prospects. Et gérer 4 réseaux manuellement, personne n'a le temps.

04 / LE QUOTIDIEN

4 problèmes concrets.

⏳

Pas le temps

Créer du contenu pour Facebook, puis l'adapter pour Instagram, LinkedIn et Google Business. C'est 2-3 heures par semaine que le courtier n'a pas.

Résultat : les comptes sont morts depuis 6 mois.
📝

Contenu non conforme

L'assurance est réglementée. On ne peut pas écrire n'importe quoi. Pas de promesses de rendement, pas de garanties absolues, mentions obligatoires.

Risque : publication non conforme = problème FSMA/ACPR.
🎨

Contenu générique

Des posts identiques pour tous les courtiers. Pas de logo, pas de couleurs du cabinet. Le client ne reconnaît pas son courtier.

Résultat : zéro engagement, zéro conversion.
⭐

Ambassadeurs gâchés

Advice identifie des clients satisfaits, mais personne ne fait rien avec. Leurs témoignages restent dans un tableau. Jamais publiés.

Coût : visibilité perdue, nouveaux clients qui ne viendront pas.

06 / LA SOLUTION

Comment Connect fonctionne.

1

L'admin crée un article

Depuis le back-office : titre, texte, image, catégorie. Templates avec variables JSON ({nom_courtier}, {ville}, {specialite}).

2

Connect génère N visuels

Un visuel unique par courtier : son logo, ses couleurs, son nom, sa ville. Pré-généré avant la date de publication.

3

Planification

Date et heure programmées. Filtrage par compagnie, branche, zone géographique. Seuls les courtiers concernés reçoivent le post.

4

Publication simultanée

Cron déclenche à la date prévue. Post le visuel personnalisé sur chaque réseau connecté de chaque courtier. Rate limiting respecté.

5

Collecte d'engagement

Chaque nuit, les métriques sont collectées : vues, likes, commentaires, partages, taux d'engagement. Dashboard dédié.

07 / 4 RÉSEAUX

4 plateformes connectées.

f

Facebook

API : Meta Graph API v18+
Auth : OAuth2 (pages_manage_posts, pages_read_engagement)
Format : Image + texte sur la page du courtier
Limites : 200 posts/heure (largement suffisant)
Métriques : Portée, likes, commentaires, partages
⚙

Instagram

API : Meta Graph API (même token que Facebook)
Auth : Même OAuth, scope instagram_basic + content_publish
Format : Image obligatoire + légende (pas de text-only)
Limites : 25 posts/jour par compte
Métriques : Impressions, likes, commentaires, saves
in

LinkedIn

API : LinkedIn Marketing API v2
Auth : OAuth 2.0 (w_member_social, r_organization_social)
Format : Image + texte sur la page entreprise
Limites : 100 posts/jour par organisation
Métriques : Impressions, clics, réactions, commentaires
G

Google Business

API : Google Business Profile API
Auth : OAuth 2.0 (business.manage)
Format : Post avec image + texte + CTA optionnel
Limites : Posts expirent après 7 jours
Métriques : Vues, clics (via Search Console)

09 / API RÉSILIENCE

Meta change tout le temps.

L'API Facebook/Instagram change plusieurs fois par an. LinkedIn aussi. Google Business Profile a ses propres caprices. Le système doit encaisser sans tomber.

PROD

Couche d'abstraction

Interface unifiée pour publier sur les 4 réseaux. Si Meta change son endpoint, on ne modifie qu'un seul fichier — pas toute l'app.

PROD

Dégradation gracieuse

Si un réseau est down, les 3 autres continuent. Le post est mis en file d'attente pour le réseau en panne. Notification au courtier : "Instagram temporairement indisponible, publication reportée".

PROD

Auto-retry intelligent

Backoff exponentiel (30s, 2min, 10min). Différenciation entre erreur temporaire (retry) et erreur permanente (notification). Pas de spam d'appels API.

13 / SÉCURITÉ OAUTH

4 providers, 200 tokens.

Chaque courtier connecte jusqu'à 4 comptes sociaux. C'est 4 tokens OAuth à gérer par courtier. À 50 courtiers, ça fait 200 tokens vivants en parallèle.

14 / GESTION DES TOKENS

Token management.

PROD

Refresh scheduling

Cron job qui vérifie l'expiration de chaque token. Refresh automatique 48h avant expiration. Pas de surprise le jour où le courtier veut publier.

PROD

Révocation handling

Si un courtier révoque l'accès depuis Facebook/LinkedIn : détection immédiate, désactivation propre du réseau, invitation à reconnecter dans l'app.

PROD

Isolation des tokens

Chaque token chiffré AES-256-GCM. RLS strict : un courtier ne peut jamais accéder aux tokens d'un autre. Même en cas de faille, isolation totale.

PROD

Dashboard connexions

Le courtier voit l'état de chaque connexion : vert (OK), orange (expire bientôt), rouge (expiré/révoqué). Bouton reconnecter en 1 clic.

15 / EDGE CASES

Quand ça ne se passe pas bien.

Les réseaux sociaux sont imprévisibles. Chaque cas limite est anticipé.

PROD

Réseau déconnecté en pleine campagne

Le courtier lance une publication multi-réseaux. Facebook tombe au milieu. Fix : Les 3 autres publient normalement. Le post Facebook passe en file d'attente. Retry auto. Notification : "Facebook publié avec 15 min de retard".

PROD

Upload image échoue

Problème réseau pendant l'upload du visuel. Fix : Retry 3x. Si échec total, fallback vers publication text-only (sauf Instagram qui exige une image → reporté + notification).

PROD

LinkedIn rate limit (100 posts/jour)

Avec 50 courtiers qui publient 2-3 fois/jour, on approche la limite. Fix : Compteur par app. Staggering des publications. Si limite atteinte : file d'attente + publication au prochain créneau libre.

PROD

Instagram : pas de text-only

Instagram n'accepte pas les publications sans image. Fix : Si le courtier n'ajoute pas d'image, génération automatique d'un visuel à partir du texte (template + couleurs du courtier). Jamais de publication échouée.

16 / MULTI-TENANT

50 courtiers × 4 réseaux.

200 connexions OAuth. Des dizaines de publications par jour. Le système doit scaler sans effort.

PROD

200 connexions OAuth gérées

50 courtiers × 4 réseaux = 200 tokens vivants. Chaque token a son cycle de vie (création, refresh, expiration, révocation). Le système gère tout automatiquement.

PROD

Publication staggering

Si 10 courtiers planifient un post à 9h00, le système étale les publications sur 9h00-9h15 pour éviter les rate limits. Chaque courtier voit "publié à 9h00" dans son calendrier.

PROD

Isolation des données

RLS strict. Chaque courtier ne voit que ses posts, ses stats, ses connexions. Vérification cross-tenant avec 5+ courtiers de test.

17 / ENGAGEMENT DATA RELIABILITY

Des données fiables.

PROD

API inconsistencies

Meta change le format des métriques régulièrement. Fix : Parsing adaptatif avec fallback. Si un champ manque, on affiche "donnée indisponible" plutôt qu'un chiffre faux.

PROD

Data gaps

Réseau down pendant la collecte des stats. Fix : Retry plus tard. Indicateur visuel "données du 15 mars manquantes". Pas de trou dans les graphiques, interpolation indiquée.

PROD

Historical backfill

Quand un courtier connecte un réseau, on récupère l'historique des 30 derniers jours (posts existants + engagement). Le dashboard n'est pas vide dès le jour 1.
